## **Commercial Engine – Policy Gates & Human Agency Doctrine**

This document defines the governing truth for how enforcement, control, and AI operate inside the Commercial Engine.

It is binding design doctrine.

---

# **1\. Foundational Principle**

This is a **Human-First, AI-in-the-Loop** system.

The system exists to:

* Protect margin

* Protect operational feasibility

* Reduce dependency on Amin

* Preserve commercial truth

* Enable structured governance

The system does **not** exist to:

* Remove human agency

* Automate negotiation authority

* Block commercial judgment

* Replace decision-makers

Humans always retain final authority.

---

# **2\. Policy Gates (Not Hard Locks)**

The system does not use “hard locks.”

It uses **Policy Gates**.

A Policy Gate is a configurable control that:

* Enforces doctrine

* Protects governance

* Preserves commercial structure

* Can be configured, relaxed, or overridden by authorized humans

Policy Gates are default safeguards — not prison bars.

---

# **3\. Enforcement Modes (Admin Controlled)**

Every Policy Gate must support the following enforcement modes:

### **1\. Enforce**

* Blocks progression by default

* Can still be overridden (if override is enabled)

* Requires reason capture

* Fully audited

### **2\. Warn**

* Displays structured warning

* Requires acknowledgement

* Does not block progression

### **3\. Off**

* Gate is disabled entirely

* No warning

* No enforcement

No gate is permanently hardcoded.

All enforcement behavior must be configurable from the Admin Panel.

---

# **4\. Override Doctrine (“Break Glass”)**

Even when a Policy Gate is set to Enforce:

Authorized roles may override.

Override requires:

* Explicit reason entry

* Optional supporting documentation (configurable)

* Timestamp

* User identity

* Rule version at time of override

Optional configuration:

* Require second approval

* Notify specific stakeholders

* Escalate based on margin threshold

Overrides are intentional and traceable.

They are not silent bypasses.

---

# **5\. No Creeping Lockdown Principle**

The system must never evolve into:

* Silent auto-rejection

* Hidden margin enforcement

* AI-driven decision substitution

* Unconfigurable compliance restrictions

* Hard-coded, non-editable commercial blocks

All gates must:

* Be visible

* Be configurable

* Be reversible

* Be auditable

No creeping automation.  
 No creeping rigidity.  
 No hidden decision logic.

---

# **6\. Admin & Governance Console (Mandatory)**

The system must include a first-class Admin Panel that controls:

## **A) Policy Gates**

* Enforcement mode (Enforce / Warn / Off)

* Override enabled/disabled

* Who can override (RBAC)

* Whether second approval is required

* What justification fields are mandatory

* Regional or BU-specific scope

## **B) Stage Controls**

* Enable/disable stages

* Modify stage sequence (within doctrine constraints)

* Adjust required fields per stage

## **C) Bot Governance**

Bots are optional assistants.

They must be:

* Explicitly created

* Explicitly configured

* Explicitly enabled

Each bot must have:

* Defined purpose

* Defined scope (Quote, Proposal, SLA, etc.)

* Defined allowed actions (Suggest-only / Draft-only / Explain-only)

* Temperature control

* Instruction set

* Tool access permissions

* Audit trail of outputs

Mandatory controls:

* Global bot kill switch

* Per-bot disable toggle

* Full output trace logging

Bots cannot:

* Approve

* Override

* Modify pricing

* Change scope

* Execute actions autonomously

AI assists.  
 Humans decide.

# **7\. Commercial Doctrine Protection**

Even with overrides allowed, the system must still preserve:

* Traceable lifecycle from Opportunity → Contract Readiness

No override may:

Truth is immutable.

## **1\) Purpose**

The TOC exists to:

* Give humans **structural control** over long commercial documents

* Support **multi-part** proposals and tender packs

* Enable clean **PDF assembly** (sections map to output order)

* Preserve **Draft vs Canon** boundaries at scale

* Avoid hidden restructuring or AI-driven document mutation

The TOC is a navigation \+ structure layer.  
 It is not an AI feature.

---

## **2\) Core Doctrine**

### **2.1 The Editor Must Support a Three-Level Hierarchy**

All commercial authoring uses:

**Workspace → Document → Section → Block**

Not “Document → Blocks only”.

This is mandatory for:

* Proposals with multiple sections

* SLAs with schedules/annexes

* Tender packs (technical \+ commercial \+ pricing schedules)

---

## **3\) Definitions**

* **Document**: A top-level authored artifact inside a Workspace (e.g., Proposal v3, SLA v1, Tender Technical Volume).

* **Section**: A TOC item within a Document (e.g., Executive Summary, Scope of Work, Commercial Model).

* **Block**: The smallest editable unit inside a Section (paragraph, table, clause, list, etc.).

* **TOC Panel**: The UI surface that displays Documents and Sections and allows human navigation and ordering.

---

## **4\) TOC Ownership and Authority**

### **4.1 Humans Own Structure**

Only humans can:

* Create documents

* Create sections

* Rename sections

* Reorder sections

* Move blocks between sections

No bot, automation, or background process may restructure TOC items.

### **4.2 AI Has Zero Structural Authority**

AI cannot:

* Create sections without explicit human action

* Reorder sections

* Merge sections

* Split sections

* Move blocks between sections

AI may draft **content**, not structure.

---

## **5\) TOC Interaction Rules**

### **5.1 Section-Level Navigation**

The TOC must support:

* Jump to section

* Expand/collapse sections

* Show section status indicators (Draft / Canon / Locked)

### **5.2 Section Ordering**

Sections must be reorderable by humans (drag/drop or move up/down).

Reordering must:

* Preserve block IDs

* Preserve block revision history

* Produce an audit event

No silent reorder.

### **5.3 Multiple Documents in a Single Pack**

A commercial “pack” may contain multiple Documents.

Example:

* Tender Technical Volume (Document A)

* Tender Commercial Volume (Document B)

* Pricing Schedule (Document C)

The TOC must allow selecting which Document is active in the editor.

---

## **6\) Draft vs Canon Rules with TOC**

### **6.1 Draft and Canon Apply at Block Level**

Blocks exist in either:

* Draft

* Canon

Draft blocks can be edited freely.

Canon blocks are immutable unless explicitly unlocked (per existing Truthpack).

### **6.2 Promotion Rules**

Nothing moves from Draft to Canon except by explicit human action.

Promotion may occur:

* Per-block (required)

* Per-section (optional later, still must be explicit and auditable)

AI cannot promote content.

## **7\) Locking Rules at Section and Document Level**

The system may support locking at:

* Block level (required)

* Section level (allowed)

* Document level (allowed)

Locking must always be:

* Human-triggered

* Audited

Locked content cannot be modified by AI or humans without unlock.

## **Block Controls, Tagging, and Tagged Block Library Doctrine**

### **1\) Block Control Requirements (Mandatory UI \+ Behavior)**

Every **Block** in Structure Mode must expose the following human controls:

**A. Select Tick Box**

* Each block has a selectable checkbox.

* Multi-select enables **bulk actions**:

  * Export selected blocks → **Draft**

  * Promote selected blocks → **Canon**

  * Save selected blocks → **Library**

* Selection does **not** change content automatically. It only enables user actions.

**B. Delete Button**

* Each block has a delete control.

* Delete must be:

  * Explicit (click)

  * Confirmable (optional confirmation toggle in Admin)

  * Audited (AuditEvent: `BLOCK_DELETED`)

* Delete must not silently remove Canon content without unlock permissions.

**C. Tag / Label Control**

* Each block supports one or more **tags** (labels).

* Tags are **metadata**, not governance.

* Tags must not trigger:

  * approvals

  * gates

  * workflow transitions

  * signals  
     unless explicitly designed later.

**D. Move Controls**

* Blocks must be movable:

  * Up/down buttons AND/OR drag handle (preferred)

* Move operations must:

  * Preserve blockId

  * Preserve revision history

  * Preserve provenance

  * Emit audit event: `BLOCK_MOVED`

---

### **2\) Bulk Export Rules (Draft / Canon)**

**A. Export to Draft**

* Selected blocks can be exported into **Draft Mode** compilation.

* This creates a Draft assembly view; it does not alter Canon.

* Export action must be explicit: “Send to Draft”.

**B. Promote to Canon**

* Selected Draft blocks can be promoted into Canon only via explicit action:

  * “Promote Selection to Canon”

* Promotion compiles content into Canon view (collapsed document), per Truthpack.

* Promotion must:

  * Record audit event `PROMOTE_TO_CANON`

  * Lock Canon by default

AI cannot bulk export or bulk promote.

Humans only.

---

### **3\) Tagged Block Library (Reusable Components)**

The system must provide a **Tagged Block Library** for reusable block snippets.

#### **3.1 What the Library Is**

* A storage area for curated, reusable blocks (e.g., standard clauses, disclaimers, scope paragraphs).

* Blocks are stored with:

  * content

  * tags

  * type (heading/paragraph/clause/table)

  * provenance metadata

  * createdBy

#### **3.2 What the Library Is Not**

* It is not Canon.

* It does not auto-inject content into documents.

* It is not AI-managed.

#### **3.3 How Blocks Enter the Library**

Only via explicit human action:

* Select block(s) → “Save to Library”

* Or “Save as Template Block”

This creates a **LibraryBlock** record.  
 No silent saving.

#### **3.4 How Blocks Are Used**

Humans may:

* Drag-and-drop Library blocks into the editor

* Or click “Insert into Draft” / “Insert into Structure”

Insertion must:

* Create a new Draft block

* Preserve original Library block as immutable reference

* Record audit event `LIBRARY_BLOCK_INSERTED`

---

### **4\) Governance & Safety Constraints**

**A. No AI Library Insertion**  
 AI cannot:

* insert blocks from library automatically

* rewrite library blocks silently

* promote library blocks to Canon

AI can suggest “this clause might help,” but insertion remains human.

**B. Audit Requirements**  
 Audit must capture:

* block selection operations (optional)

* block deletion

* block move

* tag edits

* saved to library

* inserted from library

* promoted to canon

**C. Canon Protection**  
 If a block is Canon-locked:

* delete requires unlock permission \+ reason \+ audit

* edit requires unlock permission \+ reason \+ audit

* move may be restricted (configurable) but must remain human-controlled

---

### **5\) Summary: Non-Negotiables**

* Blocks have checkbox selection

* Blocks have delete

* Blocks have tags

* Blocks move via drag handle or move buttons

* Tags are metadata only

* Selected blocks can be bulk-exported to Draft or promoted to Canon

* Tagged blocks can be saved into a library and dragged back into editor

* No AI autonomy in any of these actions
